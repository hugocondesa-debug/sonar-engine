## 3. Hypothesis ranking

Given the negative filesystem evidence + the commit message quality
(sprint-aware, brief-aligned, conventional-commits compliant), the
ranked hypotheses:

### H1 — Parallel Claude Code instance (HIGH confidence)

**Mechanism**: Hugo runs a second CC session, concurrent with the
primary sprint CC session. The second CC instance (likely orchestrated
via tmux — Hugo maintains 2+ tmux sessions during active sprints)
sees the filesystem changes, runs `git add` + `git commit` on its own
cadence. Because it runs as user `macro` with git config
`user.email=hugocondesa@gmail.com`, commits are authored as Hugo.

**Why commits lack the `Co-Authored-By` trailer**: the second CC
instance's prompt does NOT include the CLAUDE.md §5 directive (it may
run under a different system prompt, or the operator issues the
commit via `/git-commit` or similar without the trailer template).
The primary CC commits (`13dee4e` + `9f744c4`) DO have the trailer
because they follow CLAUDE.md §5 directly.

**Supports**:
- Commit message quality — sprint-aware, narrative, conventional
  scope. Matches CC authorship signature.
- Timing — five commits in ~13 minutes (20:35-20:48) is CC-throughput-plausible,
  implausible for manual typing.
- No filesystem evidence of a daemon — because there isn't one;
  commits run inside an interactive shell.
- No `Co-Authored-By` trailer — different CC instance with different
  system-prompt template.

**Against**:
- None directly. The only way to falsify this hypothesis is
  interrogating the operator on whether a second CC instance was
  active during Sprint M (operator confirmation needed).

### H2 — Scheduled Claude agent / routine (LOW confidence)

**Mechanism**: Hugo has a `/schedule`d Claude agent or `/loop`ed
routine running from claude.ai that periodically fires `git add` +
`git commit` based on diff heuristics.

**Against**:
- No `claude agent` / `claude routine` evidence on VPS
  (`~/.claude/` has no hook or routine config).
- Scheduled agents typically run from claude.ai cloud infrastructure
  and would need a local endpoint (SSH, CLI trigger) which would
  leave logs. None found in the audit.
- Sprint M retrospective does not mention operator setting up a
  routine — framed as an *observed* entity, not an *installed* one.

**Verdict**: unlikely but not disproved. Would require probing
claude.ai routine config, which is outside this VPS's scope.

### H3 — Custom inotify / shell daemon installed outside PATH (LOW confidence)

**Mechanism**: a stealth daemon started from a non-standard location
(e.g., `~/bin/auto-commit-watcher &` backgrounded via tmux
`send-keys`) watches inotify events + auto-commits.

**Against**:
- `ps auxf` shows nothing matching.
- No such script under `/home/macro/projects/sonar-engine/scripts/`
  or `~/bin/` or `/usr/local/bin/`.
- Would not explain CC-grade commit message narrative (a shell script
  would produce formulaic, non-sprint-aware messages).

**Verdict**: unlikely. Message quality alone nearly eliminates this.

### H4 — Operator manually committing via a separate terminal (NEGLIGIBLE confidence)

**Mechanism**: Hugo himself types the commits in a separate terminal
while the primary CC session is mid-sprint.

**Against**:
- Five commits in 13 minutes with full narrative bodies is
  implausible manual-typing throughput for a reviewer-mode operator
  (retrospective frames Hugo as reviewer, not coder, during sprints).
- Sprint M retrospective describes the commits as "racing against CC"
  — language inconsistent with operator self-attribution.

**Verdict**: ruled out on throughput grounds alone.

## 4. Commit-signature pattern (what a future CC should key on)

If the watcher is retained, future CC sessions can detect "this
commit was landed by the watcher" via:

1. **Trailer absence** — `git log --format="%(trailers:key=Co-Authored-By)" <SHA>` returns empty.
2. **Message style parity** — Conventional Commits, sprint-aware
   narrative body. (Indistinguishable from CC-manual by message
   content alone.)
3. **Timing proximity to file save** — commit typically lands within
   30-120 seconds of the last file save (Sprint M observed window).

Defensive pattern from Sprint M retrospective §8 Lesson #13:

```
git log --oneline -5   # run BEFORE every manual git commit
# If the expected change is already shipped, skip the manual commit.
# If not, proceed.
```

## 5. Recommendation (3 options — operator decides)

### Option A — Disable

**When to pick**: if the watcher is an experimental residual from an
earlier Phase 0-1 workflow and is no longer intentional; OR if
governance review concludes that non-reviewed commits should never
enter the historical record.

**Risk**: negligible. Disabling restores the canonical
`CC-stage → CC-commit` workflow; CC throughput drops marginally
(Sprint M's 5/9 watcher commits each save CC ~60-90s of commit
orchestration).

**How**: requires operator to identify the second CC instance /
routine / daemon and stop it. Since the VPS audit could not locate
it, the operator-side teardown would likely be: close the
second-tmux CC session OR disable a claude.ai routine, depending on
which hypothesis (H1 vs H2) turns out correct.

### Option B — Document as canonical

**When to pick**: if the watcher is intentional parallel-CC
orchestration and the compound value (no-op saved on commit
orchestration) is worth the cognitive cost of the dual-author model.

**Action**:
1. Add a `Machine Discipline § Parallel commit model` section to
   SESSION_CONTEXT explaining the watcher as canonical behavior:
   - What entity commits what (scope boundaries).
   - Why the `Co-Authored-By` trailer diverges (two CC system
     prompts with different templates — OR unify the template so
     both instances trailer-sign).
   - When CC-primary should defer vs. manually commit.
2. Update `sprint_setup.sh` to emit a `[sprint_setup] watcher: active`
   line during worktree creation (Sprint M retrospective §8
   recommendation) — so CC arranque knows upfront.
3. Extend brief format v3.2 with an optional §6.5 "Commit
   attribution model" field declaring watcher-on / watcher-off
   per-sprint.

**Risk**: medium. Canonizing a second non-reviewed commit source
increases the surface area of review (operator must trust watcher
scope-correctness without reading diffs). Historical record becomes
dual-sourced; accountability for a bad commit is ambiguous.

### Option C — Scope-limit (retain but constrain)

**When to pick**: if the watcher's compound value is worth retaining
but unreviewed commits on certain paths / branches are unacceptable.

**Action**:
1. Constrain watcher to specific paths — e.g., watcher may commit to
   `src/sonar/connectors/` + `tests/unit/` but NOT to `docs/` or
   `scripts/ops/`. CC-primary retains exclusive commit rights over
   meta-work files.
2. Constrain watcher to specific branches — e.g., watcher only
   active on `sprint-*` branches during sprints; never on `main`.
3. Enforce a `Co-Authored-By: <watcher-ID>` trailer so commit
   authorship is unambiguously sourced.
4. Expose a `WATCHER_DISABLED=1` env var that sprint_setup.sh can
   flip per-sprint for sensitive sprints (e.g., ADR edits, governance
   changes, schema migrations).

**Risk**: low. Scope-limiting keeps the compound value and adds
defensibility. Implementation cost: the mechanism behind H1 (H2 /
H3) must be identified first so the scope fence can be plumbed.
